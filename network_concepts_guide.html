TCP/IP Architecture

TCP/IP is the practical protocol suite that powers modern networks and the internet. Unlike OSI, which is a conceptual model, TCP/IP is used in real-world communication stacks. It condenses networking into four layers.

Combines OSI Application, Presentation, and Session functions. It supports user-facing services such as web access, name resolution, file transfer, remote administration, email, and application APIs.

HTTP/HTTPS, DNS, SMTP, IMAP, FTP/SFTP, SSH, DHCP, SNMP

Internet Layer

Responsible for logical addressing and routing packets across interconnected networks. It determines where packets go and how they traverse networks.

Encapsulation and Decapsulation

This is one of the most fundamental networking concepts. As data moves down the sender's protocol stack, each layer wraps the previous layer's output with its own header (and at Layer 2, a trailer). The result grows progressively larger — like nested envelopes. On the receiving side, each layer reads and strips its own header, passing the payload upward until the original application data is delivered.

Postal analogy: Think of it like mailing a letter. The data is the letter. The Transport layer puts it in an envelope with port numbers. The Network layer adds a shipping label with IP addresses. The Data Link layer places it in a delivery pouch with MAC addresses and a seal (FCS). The Physical layer converts it to electrical signals on the wire. The receiver opens each layer in reverse.

End-to-End Packet Flow

From URL to Response — the full network journey. This visual traces how a request moves from a user's browser, through network transit (routers and switches), to a remote server and back — layer by layer, hop by hop.

Key insight: At the endpoints (client and server), all 7 OSI layers are involved. At routers, only L1–L3 are processed — the router strips the L2 frame, reads the L3 IP destination, and builds a new L2 frame for the next hop. At switches, only L1–L2 are processed. The original L3 packet and L4 segment remain intact across all intermediate hops.

Core Protocols of TCP/IP Architecture

These protocols form the working language of modern networks. Expand each area for a fast, practical explanation.

HTTP / HTTPS
HTTP transfers web content. HTTPS is HTTP protected with TLS for confidentiality, integrity, and server authentication. It is foundational for websites, APIs, portals, SaaS platforms, and enterprise web applications.
DNS
The Domain Name System resolves names such as example.com into IP addresses. Without DNS, users would need to remember numeric IP addresses. It is critical, and therefore a common target for abuse, poisoning, tunneling, and redirection.

Record types: A, AAAA, CNAME, MX, NS, TXT, PTR, SOA

SMTP / IMAP / POP3
SMTP is used to send email. IMAP and POP3 are used to retrieve email, with IMAP allowing richer synchronization across devices. Security depends heavily on TLS, authentication, and anti-phishing controls.
FTP / SFTP / SSH
FTP moves files but is insecure in plaintext. SFTP runs over SSH and provides secure file transfer. SSH itself enables encrypted remote administration and secure command-line access.
DHCP
Dynamic Host Configuration Protocol automatically assigns IP addresses and other network settings such as default gateway and DNS servers to clients joining a network.
SNMP
Simple Network Management Protocol is used to monitor and manage network devices such as routers, switches, and servers. SNMPv3 adds encryption and authentication. It operates over UDP ports 161 (queries) and 162 (traps).
LDAP / LDAPS
Lightweight Directory Access Protocol is used to query and manage directory services such as Active Directory. LDAPS adds TLS encryption. Commonly used for authentication, authorization, and user/group management in enterprise environments.
TCP
Transmission Control Protocol is connection-oriented and reliable. It uses sequence numbers, acknowledgements, retransmissions, and flow control. It is used where correctness matters, such as web sessions, file transfer, and email transport.
UDP
User Datagram Protocol is connectionless and lightweight. It does not guarantee delivery, ordering, or retransmission. It is used where speed and low overhead matter, such as streaming, VoIP, DNS queries, and some real-time applications.
Ports and Sockets
Ports identify logical services on a host, while sockets represent communication endpoints. Firewalls and transport-layer controls often allow or block traffic based on port and protocol combinations.
IP (IPv4 / IPv6)
Internet Protocol handles addressing and routing across networks. IPv4 is still dominant, while IPv6 provides a vastly larger address space and improved addressing structure.
ICMP
Internet Control Message Protocol supports diagnostics and control messaging, such as ping and path testing. It is useful operationally but can also be abused for scanning and flood activity.
IPsec
IPsec protects IP traffic with encryption and authentication, often used in VPNs and secure site-to-site communication at the network layer.
Routing Protocols
Protocols such as OSPF, BGP, and EIGRP determine how routes are learned and exchanged. They are essential for enterprise and internet routing and therefore high-impact from a resilience and trust perspective.
Ethernet
Ethernet is the dominant wired LAN technology. It defines frame structures, MAC addressing behaviour, and media access characteristics for local networks.
Wi-Fi
Wi-Fi provides wireless LAN connectivity. Security depends on encryption, authentication, segmentation, and strong wireless design practices.
ARP
Address Resolution Protocol maps IPv4 addresses to MAC addresses inside a local network. Because ARP lacks strong native trust controls, it is a common source of spoofing and poisoning attacks.
MAC Addressing
MAC addresses identify network interfaces at Layer 2. Switching and access control decisions at the local network layer often depend on them.

Communication Model

Connection-Oriented vs Connectionless

Connection-Oriented (TCP, SCTP)

Session setup via three-way handshake. Guarantees delivery, ordering, and error correction. Flow control via sliding window. Graceful teardown via FIN/ACK. Use: web, email, file transfer, APIs.

Connectionless (UDP, ICMP)

No session setup — data sent immediately. No delivery guarantee, ordering, or retransmission. Minimal 8-byte header overhead. Use: DNS, VoIP, streaming, gaming, NTP.

Security Control Model

Stateful vs Stateless Controls

Stateless

Evaluates each packet independently against rules. No memory of prior traffic. Examples: simple packet-filter ACLs on routers. Faster processing but cannot distinguish legitimate replies from unsolicited inbound traffic.

Stateful

Maintains a connection state table tracking active sessions. Understands whether a packet belongs to an established session or is unexpected. Examples: stateful firewalls, NGFWs. Allows return traffic automatically; detects out-of-state packets.

How they connect: Connection-oriented protocols (TCP) work naturally with stateful controls because both track session state. Connectionless protocols (UDP) are harder for stateful devices to inspect — they typically use timeouts and heuristics to create pseudo-state entries.

Core Networking Devices by Function

Router

Forwards packets between networks and makes routing decisions based on IP addressing.

Switch

Connects devices within the same local network and forwards frames using MAC addresses.

Firewall

Filters traffic based on rules. It may work at Layer 3, Layer 4, or Layer 7 depending on design.

Load Balancer

Distributes traffic across multiple servers for performance and availability.

IDS / IPS

Detects or blocks suspicious traffic patterns and known malicious signatures or behaviours.

DNS Server

Translates domain names to IP addresses so systems can locate one another.

Proxy Server

Acts as an intermediary between clients and servers, providing caching, filtering, access control, and anonymity.

VPN Gateway

Terminates encrypted VPN tunnels, providing secure remote access or site-to-site connectivity over untrusted networks.

SIEM

Security Information and Event Management platform that aggregates logs, correlates events, and provides alerting for security operations.

Threats, Threat Actors, Vulnerabilities, Attacks, Controls, Attack Vectors and Mitigations

These terms are related but not interchangeable. Understanding the distinction is essential for risk assessment, incident response, control design, and security strategy.

Threat

Potential cause of harm

A threat is anything capable of causing damage by exploiting a weakness. It may be human, technical, environmental, or operational.

MalwareInsider misusePower failure
Threat Actor

Entity that can execute a threat

Threat actors include cybercriminals, nation-state operators, insiders, hacktivists, script kiddies, and malicious competitors.

CybercriminalInsiderAPT
Vulnerability

Exploitable weakness in a system

A vulnerability is a flaw in software, configuration, process, or design that can be exploited by a threat actor. Threat + Vulnerability = Risk. Common examples include unpatched software, misconfigured services, default credentials, and insecure code.

Unpatched CVEMisconfigurationWeak password
Attack Vector

Path used to reach the target

This is the route or mechanism used to carry out the attack. Examples include email, exposed ports, weak VPNs, vulnerable web forms, infected USB devices, and misconfigured APIs.

Attack

Actual exploitation attempt

An attack is the event where a threat actor takes action to compromise confidentiality, integrity, or availability.

PhishingDDoSARP Poisoning
Control

Safeguard that prevents, detects, corrects or deters

Controls may be preventive, detective, corrective, deterrent, or compensating. They can be administrative, technical, or physical.

FirewallMFAEDRSIEM
Mitigation

Action used to reduce likelihood or impact

Mitigations are practical risk-reduction measures such as patching, segmentation, secure configuration, rate limiting, user awareness, backups, and incident playbooks.

Defence in Depth — Layered Security Architecture

No single control can protect an entire network. Defence in depth is a strategy that layers multiple, overlapping security controls so that if one layer fails, the next layer catches the threat. It is the foundational principle behind every mature security architecture.

Core principle: Assume every individual control can be bypassed. Defence in depth means an attacker must defeat multiple independent barriers — across identity, endpoint, network, application, data, and physical layers — before achieving their objective. Each layer buys time, generates alerts, and limits blast radius.
Layer 1 — Physical Security

Protect the hardware and facilities

The first line of defence is preventing unauthorised physical access to servers, networking equipment, cabling, and data centres. If an attacker can touch your hardware, all other controls can be bypassed.

Locks & BiometricsCCTVGuardsVisitor LogsCable Shielding

800-53: PE (Physical and Environmental Protection)

Layer 2 — Perimeter Security

Control what enters and exits the network boundary

The network perimeter is where internal meets external. Firewalls, DMZs, and border routers filter inbound and outbound traffic. While perimeters alone are insufficient (hence Zero Trust), they remain essential for reducing attack surface.

FirewallDMZIDS/IPSDDoS MitigationBorder ACLs

800-53: SC (System and Communications Protection), SI (System and Information Integrity)

Layer 3 — Network Security

Segment and monitor internal traffic

Once inside the perimeter, internal segmentation limits lateral movement. VLANs, micro-segmentation, internal firewalls, and network monitoring ensure that a compromised host cannot freely reach all other systems.

Layer 4 — Endpoint Security

Protect every device that connects to the network

Laptops, servers, phones, and IoT devices are all attack targets. Endpoint protection includes antivirus, EDR, host firewalls, patch management, full-disk encryption, and device hardening.

Layer 5 — Application Security

Secure the software that processes data

Applications are the primary interface users and attackers interact with. Secure coding practices, input validation, WAFs, API gateways, and regular vulnerability scanning protect this layer.

Protect the ultimate target — the data itself

If all other layers fail, data-level controls are the last barrier. Encryption at rest and in transit, access controls, DLP, data classification, and backup strategies protect information even if systems are compromised.

Identity is the modern security perimeter. Strong authentication, multi-factor authentication, role-based access control, privileged access management, and continuous verification are critical at every layer.

Detection and response spans all layers. SIEM aggregates logs, SOC teams investigate alerts, incident response playbooks guide action, and disaster recovery ensures business continuity after a breach.

Each OSI layer has specific controls designed to protect it. This mapping helps engineers select the right control for the right layer during architecture and incident response.

Why layering matters

Single control failure ≠ total compromise

Consider a real-world attack path: an attacker sends a phishing email (bypasses perimeter), a user clicks (endpoint compromised), the malware moves laterally (network segmentation blocks it), or exfiltrates data (DLP detects and blocks). Each layer independently resists the attack even if the previous one fails.

  • Preventive controls — stop attacks before they succeed (firewalls, MFA, encryption)
  • Detective controls — identify attacks in progress (SIEM, IDS, log monitoring)
  • Corrective controls — limit damage and restore operations (incident response, backups, patching)
  • Deterrent controls — discourage attackers from attempting (security policies, legal warnings, visible cameras)
  • Compensating controls — alternative safeguards when primary controls are not feasible
Zero Trust Architecture (NIST SP 800-207)

Never trust, always verify — regardless of network location

Zero Trust assumes no implicit trust based on network perimeter or location. Every access request is verified continuously based on identity, device health, context, and policy. Zero Trust complements Defence in Depth — while layered defence provides multiple barriers, Zero Trust ensures that no entity is automatically trusted even within those barriers.

Verify Explicitly Least Privilege Assume Breach Micro-segmentation Continuous Monitoring

Practical Troubleshooting by Layer

This is how engineers isolate faults quickly in the real world: start at the right layer instead of guessing randomly.

Layer 1 — Physical

Check cable, power, interface status, optics, radio signal, physical damage.

Layer 2 — Data Link

Check MAC learning, switch ports, VLAN membership, ARP resolution, NAC policy.

Layer 3 — Network

Check IP address, subnet mask, gateway, route table, ICMP reachability.

Layer 4 — Transport

Check open ports, firewall policy, stateful inspection, TCP handshake issues.

Layer 5/6 — Session / Presentation

Check session state and token validity, TLS negotiation failures, certificate errors, encoding mismatches, and content-type issues. In practice, these symptoms typically surface at Layer 7 but originate from session or presentation logic.

Layer 7 — Application

Check application service health, DNS records, certificates, authentication, API response.

Mindset

Always map the symptom to the likely layer first. It reduces noise and speeds up root-cause analysis.
