OSI Layers Architecture

The OSI model is a conceptual framework that explains network communication as seven logical layers. Each layer has a specific responsibility. The model is not a protocol suite by itself; it is a reference architecture that helps you design, troubleshoot, secure, and explain network behaviour.

Why OSI still matters: In real enterprises, incidents are diagnosed and controlled faster when teams know which layer failed, which protocol is involved, which device is operating there, and which security control belongs at that layer.

Application Layer

Combines OSI Application, Presentation, and Session functions. It supports user-facing services such as web access, name resolution, file transfer, remote administration, email, and application APIs.

Transport Layer

Provides end-to-end communication between hosts. TCP gives reliable, connection-oriented delivery. UDP gives lightweight, connectionless delivery where speed matters more than guaranteed delivery.

Internet Layer

Responsible for logical addressing and routing packets across interconnected networks. It determines where packets go and how they traverse networks.

Covers how devices physically access the network medium and exchange frames on local segments. It blends the OSI Data Link and Physical responsibilities.

Encapsulation and Decapsulation

This is one of the most important networking concepts. As data moves down the sender stack, each layer adds its own header information. On the receiving side, those headers are read and removed in reverse order.

Sender Side

Encapsulation

Application: Data Transport: Segment Network: Packet Data Link: Frame Physical: Bits

Each layer wraps the data with control information needed for delivery, addressing, routing, and transmission.

Receiver Side

Decapsulation

Bits Frame Packet Segment Data

The receiving host processes and removes the headers layer by layer until the original application data is delivered.

End-to-End Packet Flow

A beginner truly understands networking only when they can explain how a request moves from a user device to a remote server and back again.

Step 1

User enters a URL

Application layer activity begins when the browser requests a destination such as a website URL.

Step 2

DNS resolves the name

The system queries DNS to convert the domain name into an IP address.

Step 3

TCP handshake starts

If the service uses TCP, the client and server establish a session using SYN, SYN-ACK, and ACK.

Step 4

TLS handshake occurs

For HTTPS, keys and certificates are exchanged so the session can be encrypted and authenticated.

Step 5

HTTP request is sent

The client sends the application request through the established secure transport path.

Step 6

Response returns

The server responds, the data is decapsulated, and the browser renders the content for the user.

Core Protocols of TCP/IP Architecture

These protocols form the working language of modern networks. Expand each area for a fast, practical explanation.

HTTP / HTTPS
HTTP transfers web content. HTTPS is HTTP protected with TLS for confidentiality, integrity, and server authentication. It is foundational for websites, APIs, portals, SaaS platforms, and enterprise web applications.
DNS
The Domain Name System resolves names such as example.com into IP addresses. Without DNS, users would need to remember numeric IP addresses. It is critical, and therefore a common target for abuse, poisoning, tunneling, and redirection.
AAAAACNAMEMXNSTXTPTRSOA
SMTP / IMAP / POP3
SMTP is used to send email. IMAP and POP3 are used to retrieve email, with IMAP allowing richer synchronization across devices. Security depends heavily on TLS, authentication, and anti-phishing controls.
FTP / SFTP / SSH
FTP moves files but is insecure in plaintext. SFTP runs over SSH and provides secure file transfer. SSH itself enables encrypted remote administration and secure command-line access.
DHCP
Dynamic Host Configuration Protocol automatically assigns IP addresses and other network settings such as default gateway and DNS servers to clients joining a network.
SNMP
Simple Network Management Protocol is used to monitor and manage network devices such as routers, switches, and servers. SNMPv3 adds encryption and authentication. It operates over UDP ports 161 (queries) and 162 (traps).
LDAP / LDAPS
Lightweight Directory Access Protocol is used to query and manage directory services such as Active Directory. LDAPS adds TLS encryption. Commonly used for authentication, authorization, and user/group management in enterprise environments.
TCP
Transmission Control Protocol is connection-oriented and reliable. It uses sequence numbers, acknowledgements, retransmissions, and flow control. It is used where correctness matters, such as web sessions, file transfer, and email transport.
UDP
User Datagram Protocol is connectionless and lightweight. It does not guarantee delivery, ordering, or retransmission. It is used where speed and low overhead matter, such as streaming, VoIP, DNS queries, and some real-time applications.
Ports and Sockets
Ports identify logical services on a host, while sockets represent communication endpoints. Firewalls and transport-layer controls often allow or block traffic based on port and protocol combinations.
IP (IPv4 / IPv6)
Internet Protocol handles addressing and routing across networks. IPv4 is still dominant, while IPv6 provides a vastly larger address space and improved addressing structure.
ICMP
Internet Control Message Protocol supports diagnostics and control messaging, such as ping and path testing. It is useful operationally but can also be abused for scanning and flood activity.
IPsec
IPsec protects IP traffic with encryption and authentication, often used in VPNs and secure site-to-site communication at the network layer.
Routing Protocols
Protocols such as OSPF, BGP, and EIGRP determine how routes are learned and exchanged. They are essential for enterprise and internet routing and therefore high-impact from a resilience and trust perspective.
Ethernet
Ethernet is the dominant wired LAN technology. It defines frame structures, MAC addressing behavior, and media access characteristics for local networks.
Wi-Fi
Wi-Fi provides wireless LAN connectivity. Security depends on encryption, authentication, segmentation, and strong wireless design practices.
ARP
Address Resolution Protocol maps IPv4 addresses to MAC addresses inside a local network. Because ARP lacks strong native trust controls, it is a common source of spoofing and poisoning attacks.
MAC Addressing
MAC addresses identify network interfaces at Layer 2. Switching and access control decisions at the local network layer often depend on them.

Core Networking Devices by Function

Beginners often memorize device names without understanding purpose. This block connects device type to operational role.

Router

Forwards packets between networks and makes routing decisions based on IP addressing.

Switch

Connects devices within the same local network and forwards frames using MAC addresses.

Firewall

Filters traffic based on rules. It may work at Layer 3, Layer 4, or Layer 7 depending on design.

Load Balancer

Distributes traffic across multiple servers for performance and availability.

IDS / IPS

Detects or blocks suspicious traffic patterns and known malicious signatures or behaviours.

DNS Server

Translates domain names to IP addresses so systems can locate one another.

Proxy Server

Acts as an intermediary between clients and servers, providing caching, filtering, access control, and anonymity.

WAF

Web Application Firewall inspects HTTP/HTTPS traffic at Layer 7 to block SQL injection, XSS, and other application-layer attacks.

VPN Gateway

Terminates encrypted VPN tunnels, providing secure remote access or site-to-site connectivity over untrusted networks.

SIEM

Security Information and Event Management platform that aggregates logs, correlates events, and provides alerting for security operations.

Threats, Threat Actors, Vulnerabilities, Attacks, Controls, Attack Vectors and Mitigations

These terms are related but not interchangeable. Understanding the distinction is essential for risk assessment, incident response, control design, and security strategy.

Threat

Potential cause of harm

A threat is anything capable of causing damage by exploiting a weakness. It may be human, technical, environmental, or operational.

MalwareInsider misusePower failure
Threat Actor

Entity that can execute a threat

Threat actors include cybercriminals, nation-state operators, insiders, hacktivists, script kiddies, and malicious competitors.

CybercriminalInsiderAPT
Vulnerability

Exploitable weakness in a system

A vulnerability is a flaw in software, configuration, process, or design that can be exploited by a threat actor. Threat + Vulnerability = Risk. Common examples include unpatched software, misconfigured services, default credentials, and insecure code.

Unpatched CVEMisconfigurationWeak password
Attack Vector

Path used to reach the target

This is the route or mechanism used to carry out the attack. Examples include email, exposed ports, weak VPNs, vulnerable web forms, infected USB devices, and misconfigured APIs.

Attack

Actual exploitation attempt

An attack is the event where a threat actor takes action to compromise confidentiality, integrity, or availability.

PhishingDDoSARP Poisoning
Control

Safeguard that prevents, detects, corrects or deters

Controls may be preventive, detective, corrective, deterrent, or compensating. They can be administrative, technical, or physical.

FirewallMFAEDRSIEM
Mitigation

Action used to reduce likelihood or impact

Mitigations are practical risk-reduction measures such as patching, segmentation, secure configuration, rate limiting, user awareness, backups, and incident playbooks.

Example chain

Threat → Vulnerability → Actor → Vector → Attack → Control → Mitigation

Example: The threat is a data breach. The vulnerability is an unpatched web framework. The threat actor is an external attacker. The attack vector is a vulnerable web form. The attack is SQL injection. The control is a WAF plus input validation. The mitigation includes patching, monitoring, and incident response.

Stateful vs Stateless Controls

This distinction matters because not all filters understand connection context. Some only inspect individual packets; others track session state.

Stateless

Evaluates packets independently

A stateless control looks at individual packets without remembering earlier traffic. Example: simple packet-filter ACLs.

Stateful

Tracks connection context

A stateful control understands established sessions and can make smarter decisions. Example: stateful firewalls and advanced inspection devices.

Shift Left Security, Start Left Security and Defence in Depth

Network security is not only about firewalls and monitoring. Mature organizations embed security early in design and sustain it across every layer of the environment.

Shift Left Security

Move security checks earlier into build and delivery

Shift left means bringing security activities earlier into development and delivery. Teams scan dependencies early, validate configurations in CI/CD, review code, and test security before deployment.

  • SAST, SCA, secret scanning
  • Secure code review and standards
  • Infrastructure as code and container checks
  • Security gates in pipelines
Start Left Security

Begin security at requirements, architecture and design

Start left goes one step earlier than shift left. It means security starts before code exists. It includes asset identification, trust boundary mapping, threat modeling, protocol choice, segmentation design, and control selection at architecture stage.

  • Threat modeling
  • Secure network architecture
  • Zero trust and segmentation design
  • Protocol and encryption decisions
Defence in Depth

Layer controls so one failure does not become total compromise

No single control is enough. Defence in depth means combining multiple overlapping safeguards across identity, endpoint, network, application, data, monitoring, backup, and physical layers.

  • Perimeter and internal segmentation
  • Identity, endpoint, network, and data controls
  • Detection, response, and recovery
  • Resilience against control failure
Zero Trust Architecture (NIST SP 800-207)

Never trust, always verify — regardless of network location

Zero Trust assumes no implicit trust based on network perimeter or location. Every access request is verified continuously based on identity, device health, context, and policy. Core principles: verify explicitly, enforce least privilege access, and assume breach at all times.

Verify Explicitly Least Privilege Assume Breach Micro-segmentation Continuous Monitoring

Practical Troubleshooting by Layer

This is how engineers isolate faults quickly in the real world: start at the right layer instead of guessing randomly.

Layer 1 — Physical

Check cable, power, interface status, optics, radio signal, physical damage.

Layer 2 — Data Link

Check MAC learning, switch ports, VLAN membership, ARP resolution, NAC policy.

Layer 3 — Network

Check IP address, subnet mask, gateway, route table, ICMP reachability.

Layer 4 — Transport

Check open ports, firewall policy, stateful inspection, TCP handshake issues.

Layer 5/6 — Session / Presentation

Check session state and token validity, TLS negotiation failures, certificate errors, encoding mismatches, and content-type issues. In practice, these symptoms typically surface at Layer 7 but originate from session or presentation logic.

Layer 7 — Application

Check application service health, DNS records, certificates, authentication, API response.

Mindset

Always map the symptom to the likely layer first. It reduces noise and speeds up root-cause analysis.

Industry Frameworks and Standards

The concepts in this guide — threats, controls, layers, and defence strategies — map directly to industry frameworks that organisations use for governance, compliance, and operations.

NIST Cybersecurity Framework (CSF)

A voluntary framework organised around six functions: Govern, Identify, Protect, Detect, Respond, and Recover. Widely adopted across industries as the baseline for cybersecurity programmes.

NIST CSF 2.0

ISO/IEC 27001 & 27002

International standards for information security management systems (ISMS). ISO 27001 defines requirements; ISO 27002 provides implementation guidance with 93 controls across organisational, people, physical, and technological domains.

ISO 27001ISO 27002

MITRE ATT&CK

A knowledge base of adversary tactics, techniques, and procedures (TTPs) based on real-world observations. Used for threat modelling, red teaming, detection engineering, and SOC maturity assessments.

ATT&CK Matrix

CIS Controls

A prioritised set of 18 security controls (v8) that map safeguards to implementation groups (IG1, IG2, IG3) based on organisational maturity. Highly actionable and often used alongside NIST CSF.

CIS v8
