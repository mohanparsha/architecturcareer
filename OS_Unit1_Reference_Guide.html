What an operating system is

An operating system is system software that sits between hardware and users, manages hardware resources, and provides the environment in which application programs execute.

User viewpoint
Resource allocator
Kernel
System program
Middleware
  • User viewpoint: convenience, ease of use, responsiveness.
  • System viewpoint: CPU, memory, storage, and I/O are limited resources that must be allocated efficiently.
  • Kernel: the one program always running, with full hardware access.
  • System programs: shells, compilers, file managers.
  • Middleware: service layers such as databases, graphics, and multimedia frameworks.
USERS │ uses ▼ APPLICATION PROGRAMS │ invoke ▼ OPERATING SYSTEM ├─ Process Scheduler ├─ Memory Manager ├─ File / I-O Manager ├─ Protection & Security └─ Communication / IPC │ manages ▼ HARDWARE
Restaurant analogy:

Hardware is the kitchen equipment. The OS is the restaurant manager coordinating tools, timing, and safety. Applications are the chefs producing output. Users are the customers placing orders without touching the equipment directly.
Four goals of the OS

Execute user programs, make the system convenient, use hardware efficiently, and provide a stable, safe, secure environment.
Why the OS is needed

Abstraction hides hardware complexity. Resource sharing prevents chaos. Protection prevents unauthorized access. Convenience improves usability. Efficiency ensures scarce resources are used well.

Chapter 1.4

Boot, multiprogramming, multitasking, dual mode

  • Bootstrap / firmware starts first and loads the kernel.
  • Multiprogramming keeps multiple jobs in memory to avoid CPU idle time.
  • Multitasking / time-sharing rapidly switches CPU among jobs to improve responsiveness.
  • Dual mode separates restricted user mode from privileged kernel mode.
  • Hardware timer prevents CPU monopoly and enables preemption.

Protection vs security

Protection asks who can access what inside the system. Security asks how the system resists unauthorized access, exploitation, and abuse.
AspectProtectionSecurity
ScopeInternal access controlDefense against internal and external attacks
GoalControlled sharing of resourcesPrevent compromise, misuse, or breach
MechanismsUID/GID, ACLs, permissionsAuthentication, encryption, sandboxing, patching
Security mechanisms in detail

UID/GID: Every user/group has a unique numeric ID; every process runs with its owner's UID. OS checks ID against permissions.
ACLs: Per-file list of (user/group, permissions) pairs — e.g., alice=rw, bob=r, others=none.
Privilege Escalation: sudo lets a user run commands as root after password verification.
Authentication: Verifying identity — passwords, biometrics (fingerprint, face), hardware tokens (YubiKey), multi-factor (MFA).
Cloud layer mapping

IaaS gives you the VM. PaaS gives you the runtime platform. SaaS gives you the finished application.

Virtualization types and distributed models

  • Type 1 Hypervisor: runs directly on hardware. Example: VMware ESXi, Xen.
  • Type 2 Hypervisor: runs on top of a host OS. Example: VirtualBox, VMware Workstation.
  • VM benefits: Isolation (crash in one VM ≠ crash in others), resource efficiency (multiple VMs per server), portability (live migration), dev/testing (many OSes without separate hardware), snapshots (instant rollback).
  • Network OS: machines remain visibly separate and are accessed explicitly.
  • Distributed OS: multiple machines appear as one unified system.
  • Client–Server: clients request, servers respond.
  • P2P: each node can act as both client and server.
Hard vs Soft Real-Time

Hard real-time: Missing a deadline is catastrophic. Examples: pacemakers, ABS brakes, aircraft control systems.
Soft real-time: Missing a deadline degrades quality but is not fatal. Examples: video streaming (dropped frames), VoIP audio.

System call implementation path

Portability
Convenience
Error handling
  1. User program calls API wrapper such as read().
  2. Wrapper places syscall number and parameters into registers or memory.
  3. Wrapper executes trap/syscall instruction.
  4. CPU flips into kernel mode.
  5. Kernel validates arguments and permissions.

ConceptMeaningWhy it matters

Policy vs mechanismChangeable decision vs fixed implementationFlexibility without redesigning mechanisms
Why applications are OS-specific

Different system call interfaces, binary formats, libraries/APIs, and calling conventions prevent direct execution across OSes.
How cross-platform compatibility IS achieved

1. Interpreted languages: Python/Java run on an OS-specific interpreter/JVM — program code is portable.
2. Cross-compilation: Same source compiled into separate binaries per OS (e.g., Chromium → .exe, ELF, .app).
3. Compatibility layers: WSL translates Linux syscalls to Windows NT calls; Wine does the reverse.
4. Containers / VMs: Package the app with its OS environment (Docker runs Linux ELF on any Linux kernel host).
OS implementation language

Modern OSes are mostly C/C++ — portable and performant. Some assembly is still needed for CPU-specific tasks (boot sequence, context switch entry point, interrupt vectors). Linux: ~95% C, ~5% assembly. Windows NT: primarily C/C++.
OS structure models

Monolithic is fast, layered is clean, microkernel is safer but IPC-heavy, modular kernels load features dynamically, and hybrid systems combine ideas for production trade-offs.
Hybrid-system note:

Modern production operating systems are rarely pure. macOS combines Mach and BSD layers, Windows NT is hybrid/modular in practice, and Android uses a Linux kernel with higher user-space runtime layers.

Program, process, address space, PCB

  • Program: passive file on disk.
  • Process: active execution instance with memory, state, and resources.
  • Multiple processes can share the same program text while maintaining separate dynamic state.

PCB field map

  • PID, PPID, UID: e.g., PID=4521, PPID=1 (init), UID=1000 (student)
  • Process state: NEW / READY / RUNNING / WAITING / TERMINATED
  • Program counter: Address of next instruction, e.g., PC = 0x00401A3C
  • CPU registers: All general-purpose, FP, SIMD registers (~100–200 bytes per process)
  • Scheduling info: Priority level, CPU time consumed, quantum remaining
  • Memory-management info: Pointer to page table, base/limit registers
  • Accounting info: CPU time used, elapsed time, start timestamp
  • Open files / I/O status: e.g., fd[0]=stdin, fd[3]=homework.txt, fd[4]=socket(TCP)
  • Signal handlers
  • Parent/child links

Analogy:

The PCB is the process dossier. If the process is paused, the OS stores everything needed to resume it exactly where it left off.
Chapter 3.2–3.4

Scheduling, dispatcher, context switching, fork–exec, IPC

  • CPU scheduler decides which ready process runs next.
  • Dispatcher performs the handoff and restores CPU state.
  • Context switching is overhead but enables multitasking.
  • fork() duplicates the process; exec() replaces the process image.
  • IPC uses shared memory or message passing.
TRIGGER: timer expiry or blocking call ▼ Save outgoing state to PCB ▼ Scheduler selects next READY process ▼ Restore incoming PCB state ▼ Return to user mode
Turnaround Time = Finish Time − Arrival Time
Waiting Time = Finish Time − Arrival Time − Burst Time
Response Time = First CPU Allocation Time − Arrival Time
CPU Utilisation = (CPU Active Time / Total Time) × 100%
Throughput = Number of Processes Completed / Time Period
SJF Burst Prediction (Exponential Average): τn+1 = α × tn + (1 − α) × τn

Context-switch overhead factors

  • Register save/restore
  • TLB flush or partial invalidation
  • CPU cache eviction and cold-cache restart
  • Page-table or memory-map switch
Practical example (modern Intel/AMD, ~2μs per switch): 50 processes, quantum = 10ms → 5,000 switches/sec → 10ms/sec → ~1% overhead ✓ 50 processes, quantum = 1ms → 50,000 switches/sec → 100ms/sec → 10% overhead ✗
Modern optimization: Process-Context Identifiers (PCID) tag TLB entries per process, avoiding full flush on switch.
Mode switch vs context switch

Mode switch changes privilege level for the same process. Context switch changes which process runs.
IPC comparison

Shared Memory is zero-copy: both processes map the same physical RAM into their virtual address spaces — no data is ever copied. This makes it the fastest IPC method for large same-machine data exchange. However, it requires explicit mutex or semaphore synchronization to prevent race conditions.

Message Passing (pipes, sockets, queues) is safer: the OS manages the channel, preventing race conditions. It works across a network but incurs system-call and kernel buffer copy overhead per message.
Zombie, orphan, daemon

A zombie is a terminated child awaiting wait() cleanup. An orphan is a child whose parent exits first. A daemon is a long-running background service process.

High-Value System Flows

Boot sequence

Power ON → BIOS / UEFI firmware starts → POST checks hardware health → Bootloader located (for example, GRUB) → Kernel loaded into memory → Kernel initializes memory, process tables, file systems → init / systemd starts core services → Shell / GUI launched for user interaction

System call path

User code → API wrapper → trap/syscall → kernel handler → service execution → return value → user resumes

Interrupt handling

Device event → interrupt line → CPU saves state → vector lookup → ISR → state restore → continue

Shell command execution

Parse command → fork() → child exec() → binary loads → runs → exit() → parent wait() → prompt returns

fork–exec–wait lifecycle

fork() duplicates parent → child path begins → exec() replaces child image → child finishes → wait() reaps zombie

How everything connects — The Big Picture

The Full Lifecycle: From Power-On to Multitasking 1. POWER ON → BOOTSTRAP (BIOS/UEFI) loads OS kernel into RAM 2. Kernel initializes KERNEL DATA STRUCTURES (linked lists for processes, queues for scheduling, hash tables for fast lookup, bitmaps for free-block tracking) 3. User logs in via CLI (shell) or GUI → USER & OS INTERFACE 4. User launches 'python3 script.py' → Shell calls fork() + exec() → SYSTEM CALLS → LINKER/LOADER maps the program into RAM → New PROCESS created with its own PCB, memory layout, state = READY 5. Scheduler dispatches → Process runs in USER MODE (mode bit = 1) [DUAL MODE OPERATION — OS is protected from user code] 6. Process calls open('data.txt') → TRAP instruction fires → CPU switches to KERNEL MODE (mode bit = 0) → OS kernel handler services the open() request, returns file descriptor → SYSRET → CPU returns to USER MODE, process continues 7. Timer interrupt fires (process has used its quantum) → CPU enters KERNEL MODE → CONTEXT SWITCH begins → Save outgoing process state to its PCB → SCHEDULER selects next READY process (using FCFS / RR / Priority / SJF) → Load next process's PCB, restore its registers, switch page table → CPU returns to USER MODE — new process runs 8. This cycle repeats THOUSANDS of times per second → MULTITASKING: multiple processes appear to run simultaneously 9. Processes communicate via IPC (shared memory or message passing) 10. Process calls exit() → TERMINATED → parent calls wait() → PCB cleaned up All of this is enabled by: Hardware interrupts + Dual mode + Timer + Scheduling + IPC

Core Control Model of the Operating System

These control concepts explain how the OS stays in charge of the machine, protects hardware, and enforces fairness.

Dual Mode Operation

  • User Mode: restricted execution for applications
  • Kernel Mode: full privilege for OS code
  • The hardware mode bit is changed only by the CPU during trap, interrupt, exception, or return operations.

Hardware Timer

  • The OS loads a timer before giving CPU control to a process.
  • When the timer expires, an interrupt forces control back to the OS.
  • This prevents infinite loops from monopolizing the CPU.

Interrupt Vector Table

  • Maps interrupt numbers to interrupt handler addresses.
  • Lets the CPU jump directly to the correct ISR.
  • Essential for fast, deterministic interrupt handling.
USER MODE (restricted) │ trap / interrupt / exception ▼ KERNEL MODE (privileged) │ IRET / SYSRET ▼ USER MODE Only hardware changes the mode bit.
CPU receives interrupt signal ▼ Finishes current instruction ▼ Saves current state (registers, PC) to stack ▼ Looks up interrupt vector table ▼ Finds handler address (ISR) ▼ Jumps to ISR (kernel mode) ▼ ISR completes ▼ Restores state and returns to user mode

SJF view (same process set)

Order by shortest burst first: P2(4) → P4(5) → P1(8) → P3(9) Gantt: | P2 0–4 | P4 4–9 | P1 9–17 | P3 17–26 | Turnaround: P2=4 P4=9 P1=17 P3=26 Waiting: P2=0 P4=4 P1=9 P3=17 Average Waiting = 7.50 ms ← optimal Average Turnaround = 14.00 ms SJF gives the mathematically minimum average waiting time for this arrival set.

SJF view — Priority process set

Process set (for Priority comparison): P1(burst=6, pri=2), P2(burst=2, pri=4), P3(burst=8, pri=1), P4(burst=3, pri=3) SJF order (shortest burst first): P2(2) → P4(3) → P1(6) → P3(8) Gantt: | P2 0–2 | P4 2–5 | P1 5–11 | P3 11–19 | Turnaround: P2=2 P4=5 P1=11 P3=19 Waiting: P2=0 P4=2 P1=5 P3=11 Average Waiting = 4.50 ms ← optimal Average Turnaround = 9.25 ms

Priority scheduling — key lesson

Use a redesigned process set to show the difference from SJF:

P1(burst=6, priority=2), P2(burst=2, priority=4),
P3(burst=8, priority=1), P4(burst=3, priority=3)

Priority order (1=highest): P3 → P1 → P4 → P2
Gantt: | P3 0–8 | P1 8–14 | P4 14–17 | P2 17–19 |

Avg Waiting = 9.75 ms (vs SJF = 4.50 ms)

Key lesson: Priority promotes P3 (the longest job!) because it has the highest priority. P2 (shortest burst) runs last. This is exactly the starvation risk — solved by aging.

Teaching interpretation

Key lesson:

FCFS can outperform Round Robin on average waiting time for some workloads, but Round Robin is still preferred in interactive systems because it improves fairness and first response.

Common Exam Traps — Do Not Confuse

Multiprogramming vs Multitasking

Utilization vs responsiveness.
Synchronous vs Asynchronous

Trap and exception are synchronous — triggered by the CPU executing an instruction. Hardware interrupts are asynchronous — triggered by external devices independently of CPU execution.
Trap vs Exception

Intentional syscall vs accidental program fault.
Interrupt vs Trap

External hardware signal vs internal deliberate software entry.
Mode Switch vs Context Switch

Privilege change vs process change.
Program vs Process

Passive file vs active execution state.
Kernel vs Shell

Kernel manages hardware; shell interprets user commands.

System Programs, Shell, and Kernel Clarification

System Programs

  • File manipulation programs
  • Status information programs
  • File modification programs
  • Programming language support
  • Program loading and execution tools
  • Communication programs
  • Background services / daemons

Shell vs Kernel

  • Shell: user-facing command interpreter
  • Kernel: privileged core managing hardware and resources
  • The shell requests services; the kernel actually performs privileged operations.

Open vs Closed Perspective

  • Open systems emphasize portability and standard interfaces.
  • Closed systems are tightly coupled to one environment or vendor stack.
  • This helps explain why APIs, ABIs, and portability matter.

Unit 1 Glossary

Compact, exam-friendly definitions for the most important Unit 1 terms.

Core system terms

Operating System: System software that manages hardware and provides services to applications.
Kernel: Core OS component always running with full privilege.
System Program: Supporting software such as shells, compilers, and file managers.
Middleware: Additional service layer above the kernel used by applications.

Protection and control terms

Interrupt: Hardware signal needing attention.
Non-Maskable Interrupt (NMI): Cannot be disabled or delayed; used for critical hardware errors (e.g., memory parity).
Maskable Interrupt: Can be temporarily disabled by the CPU; used for most device I/O. OS disables them during critical sections.
Trap: Intentional software entry into the OS.
Exception: Unintentional error condition during execution.
Mode Bit: Hardware bit marking user or kernel mode.
Privileged Instruction: Instruction allowed only in kernel mode.

Build and execution terms

Object File: Intermediate machine-code file with unresolved symbols.
Linker: Combines object files and libraries into an executable.
Loader: OS component that reads executable into memory and starts it.
Relocation: Adjusting addresses to actual loaded memory locations.

Process terms

Process: Program in execution.
PCB: Data structure storing process state and control info.
Dispatcher: Component performing the actual CPU handoff.
Preemption: Forced CPU removal from a running process.
Quantum: Fixed CPU slice in Round Robin.
Zombie Process: terminated child awaiting wait() cleanup.
Orphan Process: child whose parent exits first.
Daemon: long-running background service process.

IPC and concurrency terms

Shared Memory: Zero-copy IPC — processes map the same physical RAM into their virtual address spaces and read/write directly. Fast but needs synchronization.
Message Passing: IPC through OS-managed channels (pipes, sockets, queues). Safe and network-capable but incurs copy overhead.
Mutex: Mutual exclusion lock — allows only one process/thread into a critical section at a time.
Semaphore: Counter-based synchronization primitive; wait() decrements (blocks if 0), signal() increments.
Race Condition: Outcome depends on non-deterministic timing of concurrent access to shared data.

Scheduling and performance terms

Turnaround Time: Finish time minus arrival time.
Waiting Time: Time spent waiting in ready queue (= Turnaround − Burst).
Response Time: Time until first CPU service.
Convoy Effect: Short jobs wait behind a long job in FCFS — like a slow truck blocking a highway.
Starvation: A process waits indefinitely because higher-priority processes always take the CPU first.
Aging: Gradually increasing the priority of waiting processes over time to prevent starvation.
Scheduler: OS component that decides which READY process gets the CPU next.
Time Quantum: Fixed maximum CPU slice per process in Round Robin scheduling.
Context: Complete CPU state of a process: all registers, program counter, stack pointer, page table pointer.
Context Switch: Saving outgoing process context to its PCB; loading incoming process context from its PCB.
TLB: Translation Lookaside Buffer — CPU hardware cache for virtual→physical address translation. Flushed on context switch.

Ch1 hardware and environment terms

Bootstrap / Firmware: ROM-based program (BIOS/UEFI) that initialises hardware and loads the OS kernel on power-on.
ISR: Interrupt Service Routine — OS kernel code that handles a specific interrupt type.
DMA: Direct Memory Access — device controller transfers data directly to/from RAM without per-byte CPU involvement; CPU receives one completion interrupt.
Storage Hierarchy: Registers → Cache → RAM → SSD → HDD → Tape. Speed falls, capacity rises at each level.
Multiprogramming: Multiple jobs in memory simultaneously; CPU switches to another when one waits for I/O. Goal: utilisation.
Multitasking: CPU switches rapidly among jobs (every 10–100 ms) so all appear simultaneous. Goal: responsiveness.
Dual Mode: Hardware-enforced split between User Mode (restricted) and Kernel Mode (full privilege).
Hypervisor: Software that creates and manages virtual machines. Type 1 = bare-metal (VMware ESXi, Xen). Type 2 = hosted (VirtualBox).

Ch2 design and execution terms

System Call: Programmatic interface for user programs to request OS kernel services; causes a trap into kernel mode.
API: High-level library wrapper around system calls; portable and programmer-friendly (POSIX, Win32).
Static Linking: Library code copied into the executable at link time — self-contained binary, no external dependencies at runtime.
Dynamic Linking: Libraries (.so / .dll) loaded at runtime and shared in memory across processes — smaller binary.
Policy: The WHAT/WHEN decision — e.g., which scheduling algorithm to use. Changeable.
Mechanism: The HOW implementation — e.g., context-switch hardware. Fixed; independent of policy.
Process State: Current condition of a process: New, Ready, Running, Waiting, or Terminated.
fork(): System call creating a child process as an exact copy of the parent process.
exec(): System call replacing the current process image with a new program binary.
