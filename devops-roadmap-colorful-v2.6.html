⚡ What is DevOps?

The culture, practices, and tools that unite Development and Operations

DevOps is a set of cultural philosophies, practices, and tools that combines software development (Dev) and IT operations (Ops). The goal is to shorten the software development lifecycle and deliver high-quality software continuously. It is not a job title, a single tool, or a technology — it is a mindset and a way of working.

Traditionally, developers would write code and hand it off to a separate operations team to deploy. This created silos, communication gaps, and slow release cycles. DevOps eliminates these silos by encouraging teams to work together throughout the entire lifecycle — from planning and coding to testing, deployment, and monitoring.

Core Principles

🤝 Collaboration

Breaking down silos between development and operations teams.

🤖 Automation

Automating testing, deployment, and infrastructure provisioning to reduce human error.

🔄 CI/CD

Continuous Integration merges code frequently; Continuous Delivery/Deployment automates releases.

📊 Monitoring & Feedback

Continuously monitoring production systems and using feedback loops to improve.

📝 Infrastructure as Code

Managing infrastructure through code rather than manual processes.

🛡️ Security Integration

Embedding security checks throughout the pipeline, not just at the end.

🛡️ What is DevSecOps?

Security integrated into every phase of the software development lifecycle

DevSecOps extends the DevOps philosophy by integrating security practices into every phase of the development lifecycle, rather than treating security as a separate, final step. The traditional "security gate" model — where vulnerabilities discovered late were expensive to fix — is replaced by "shifting security left".

SAST Static Analysis

Scan source code for vulnerabilities before deployment (Semgrep, SonarQube)

DAST Dynamic Testing

Test running applications for vulnerabilities at runtime (OWASP ZAP, Nuclei)

SCA Composition Analysis

Scan dependencies and container images for known CVEs (Trivy, Grype)

IaC Scan Infra Security

Detect IaC misconfigurations before deployment (Checkov, Trivy)

Secrets Detection

Prevent hardcoded secrets from reaching repos (Gitleaks, TruffleHog)

IAM Access Control

Manage secrets and access with HashiCorp Vault, Kubernetes RBAC

Where every tool fits — the 8-stage pipeline that defines modern DevOps

How to read this: Code flows left to right through the pipeline. Each stage has tools you will learn in this roadmap. Security tools (Semgrep, Trivy, ZAP, Gitleaks) are integrated across Test and Release — this is what makes it DevSecOps, not just DevOps. The pipeline is a continuous loop: Monitor feeds back into Plan.
♻️ Continuous Integration (CI)

Code → Build → Test: Every code change is automatically built and tested. Catches bugs early before they reach production.

🚀 Continuous Delivery/Deployment (CD)

Release → Deploy: Automatically prepare releases (delivery) or deploy directly to production (deployment). Reduces manual handoffs.

🛡️ Shift-Left Security

Test → Release: Security scanning runs automatically in the pipeline — SAST on code, SCA on dependencies, DAST on running apps, secrets detection on commits.

📊 Observability Loop

Monitor → Plan: Metrics, logs, and alerts from production feed back into planning. SLIs/SLOs track reliability. Incidents drive improvements.

🧱 Prerequisites & Setup

What you need before starting your DevOps journey

Hardware Requirements

💻 Minimum Specs
  • 8 GB RAM
  • Intel i5 / AMD Ryzen 5
  • 256 GB SSD (50 GB free)
  • Stable internet connection
🚀 Recommended Specs
  • 16 GB RAM
  • Intel i7 / AMD Ryzen 7
  • 512 GB SSD
  • Broadband connection
Low-spec machines: If you have only 8 GB RAM, close unnecessary apps when running containers and use cloud-based labs like Killercoda or Play with Docker for heavier workloads.

Foundation Knowledge (from your CSE Degree)

🖥️ Operating Systems

Process management, memory, file systems → translates directly to Linux administration

🌐 Computer Networks

OSI model, TCP/IP, DNS, HTTP → foundation for networking in DevOps

💾 Databases

SQL, normalization, backups → managing databases in production systems

🐍 Programming (any language)

Logic, data structures, APIs → Python scripting & automation

🔐 Basic Security

Encryption, authentication, access control → DevSecOps foundation

📐 Software Engineering

SDLC, Agile, version control concepts → CI/CD pipeline design

Software to Install Before Starting

  • Ubuntu 24.04 LTS — as dual-boot or inside VirtualBox/VMware
  • VS Code — with extensions for YAML, Docker, Kubernetes, Python
  • Git — and create a GitHub account
  • Python 3.x — pre-installed on Ubuntu; verify with python3 --version
  • Docker Desktop — or Docker Engine on Linux

⚠️ Common Beginner Mistakes

Pitfalls that delay progress — avoid these and you'll move 2x faster

❌ Skipping Linux Fundamentals

Jumping straight to Docker or Kubernetes without understanding Linux commands, file permissions, and processes. Every DevOps tool runs on Linux.

✅ Spend the full 2 months on Phase 1. Do everything in the terminal.
❌ Learning Tools Before Concepts

Memorizing Jenkins pipeline syntax without understanding what CI/CD is, or learning Terraform commands without understanding infrastructure.

✅ Always understand the "why" before the "how". Read the concept, then use the tool.
❌ Tutorial Hell — Watching, Not Building

Watching 50 hours of YouTube tutorials without building a single project. Passive learning creates an illusion of competence.

✅ For every hour of watching, spend 2 hours building. Every phase has a milestone — build it.
❌ Chasing Every New Tool

Trying to learn Docker, Kubernetes, Terraform, Ansible, Jenkins, ArgoCD, Prometheus, and 10 more tools simultaneously.

✅ Follow the phased roadmap. Master one tool per category before exploring alternatives.
❌ Ignoring Documentation Skills

Building projects but not writing README files, architecture diagrams, or documenting decisions. Employers check your GitHub profile.

✅ Document every project with a clear README, architecture diagram, and setup instructions.
❌ Not Using Git for Everything

Writing scripts, configs, and notes locally without version control. Losing work. Not building a GitHub history.

✅ From Day 1, push every script, config file, and exercise to a GitHub repo.
❌ Prioritizing Certs Over Projects

Spending months preparing for certifications without building hands-on projects. Certs validate skills — they don't create them.

✅ Build projects first. Pursue certifications after completing the relevant phase.
❌ Studying in Isolation

Not joining communities, not asking questions, not sharing progress. The DevOps field is highly collaborative.

✅ Join communities from Day 1. Ask questions. Share what you learn. Help others.

📖 Glossary of Terms

75+ DevOps & DevSecOps terms defined — organized by category for quick reference

Core DevOps Concepts
DevOpsCulture and practices combining Development and Operations for faster, reliable software delivery
DevSecOpsDevOps with security integrated into every phase of the development lifecycle ("shift-left security")
Shift LeftMoving testing and security earlier in the development process to catch issues sooner and cheaper
AutomationUsing scripts and tools to perform tasks without manual intervention — the core principle of DevOps
IdempotencyRunning an operation multiple times produces the same result — critical for Ansible, Terraform, and reliable automation
Immutable InfrastructureServers are never modified after deployment; changes require replacing with a new version entirely
Configuration DriftWhen a server's actual state diverges from its defined configuration — IaC tools prevent this
ToilManual, repetitive operational work that can and should be automated (SRE term)
CI/CD & Pipeline
CIContinuous Integration — automatically building and testing code every time a developer pushes changes
CD (Delivery)Continuous Delivery — code is automatically prepared for release but requires manual approval to deploy
CD (Deployment)Continuous Deployment — code is automatically deployed to production without manual approval
PipelineAutomated sequence of stages (build → test → deploy) that code passes through from commit to production
ArtifactA built output (Docker image, JAR, binary) produced by a CI pipeline and stored in a registry
GitOpsUsing Git as the single source of truth for infrastructure and deployments — changes via PRs, not manual commands
Blue/GreenDeployment strategy running two identical environments; traffic switches from old (blue) to new (green) for zero downtime
Canary DeployGradually routing a small percentage of traffic to a new version before full rollout
RollbackReverting to a previous known-good version when a deployment fails or causes issues
Containerization & Orchestration
ContainerLightweight, isolated package containing an application and all its dependencies — runs identically everywhere
Docker ImageRead-only template used to create containers — built from a Dockerfile with layered filesystem
DockerfileText file with instructions to build a Docker image — defines base image, dependencies, and startup command
Docker ComposeTool for defining and running multi-container applications using a YAML configuration file
Container RegistryStorage for Docker images (Docker Hub, AWS ECR, GHCR) — like GitHub but for container images
Kubernetes (K8s)Open-source container orchestration platform — automates deployment, scaling, and management of containers
PodSmallest deployable unit in Kubernetes — one or more containers that share networking and storage
DeploymentK8s object that manages a set of identical Pods — handles scaling, updates, and rollbacks
ServiceK8s object that exposes Pods to network traffic — provides stable IP and DNS name for a group of Pods
IngressK8s object that manages external HTTP/HTTPS access to Services — handles routing, SSL, and load balancing
HelmPackage manager for Kubernetes — bundles K8s manifests into reusable, versioned charts with configurable values
RBACRole-Based Access Control — restricts who can do what in Kubernetes (or any system) based on assigned roles
Cloud & Infrastructure
VPCVirtual Private Cloud — isolated network within a cloud provider where you launch resources (AWS, GCP, Azure)
EC2Elastic Compute Cloud — AWS virtual servers (instances) that you can start, stop, and scale on demand
S3Simple Storage Service — AWS object storage for files, backups, static assets, and Terraform state
IAMIdentity and Access Management — controls who can access what in AWS (users, roles, policies, permissions)
RDSRelational Database Service — managed database (PostgreSQL, MySQL) on AWS with automated backups and scaling
Auto ScalingAutomatically adjusting the number of running servers based on demand — scales up under load, down when idle
Load BalancerDistributes incoming traffic across multiple servers to ensure availability and prevent overload (ALB, NLB)
Free TierAWS offers 12 months of limited free access to core services — essential for learning without cost
Infrastructure as Code (IaC)
IaCInfrastructure as Code — defining and provisioning infrastructure through code files instead of manual setup
TerraformHashiCorp's IaC tool — uses HCL language to provision cloud infrastructure across AWS, GCP, Azure, and more
HCLHashiCorp Configuration Language — the declarative language used to write Terraform configurations
State FileTerraform's record of what infrastructure it has created — maps your code to real cloud resources
Terraform ModuleReusable, self-contained package of Terraform configurations — like a function for infrastructure
OpenTofuOpen-source fork of Terraform (Linux Foundation) — created after HashiCorp changed Terraform's license to BSL
AnsibleAgentless configuration management tool — uses YAML playbooks to configure servers via SSH
PlaybookAnsible YAML file defining tasks to execute on remote servers — the main unit of Ansible automation
Monitoring & Observability
ObservabilityAbility to understand a system's internal state from its external outputs — metrics, logs, and traces
PrometheusOpen-source metrics collection and alerting system — pulls metrics from targets at defined intervals
GrafanaVisualization platform — creates dashboards from Prometheus, Loki, and other data sources
LokiLog aggregation system by Grafana Labs — indexes log labels (not content), pairs with Promtail for collection
PromQLPrometheus Query Language — used to query metrics and create alerting rules and dashboard panels
SLIService Level Indicator — a quantitative measure of service performance (e.g., request latency, error rate)
SLOService Level Objective — target value for an SLI (e.g., "99.9% of requests under 200ms")
SLAService Level Agreement — contractual commitment between provider and customer with consequences if not met
Error BudgetAmount of allowed unreliability (100% − SLO) — when exhausted, new feature releases are frozen until reliability improves
AlertmanagerPrometheus component that handles alert routing, grouping, silencing, and notification (Slack, email, PagerDuty)
Security & DevSecOps
SASTStatic Application Security Testing — scans source code for vulnerabilities without running it (Semgrep, SonarQube)
DASTDynamic Application Security Testing — tests running applications for vulnerabilities at runtime (OWASP ZAP, Nuclei)
SCASoftware Composition Analysis — scans dependencies and container images for known CVEs (Trivy, Grype)
CVECommon Vulnerabilities and Exposures — standardized identifier for publicly known security vulnerabilities (e.g., CVE-2021-44228)
OWASP Top 10List of the 10 most critical web application security risks — the baseline security standard every developer should know
Secrets ManagementSecurely storing and accessing credentials, API keys, and tokens — never hardcode secrets (use Vault, K8s Secrets)
Least PrivilegeGranting only the minimum permissions needed to perform a task — foundational security principle
Zero TrustSecurity model that trusts nothing by default — every request is verified regardless of origin or network location
OPAOpen Policy Agent — policy engine for enforcing rules across K8s, APIs, and infrastructure as code
CIS BenchmarksCenter for Internet Security best-practice configuration standards for operating systems, cloud, and applications
PCI-DSSPayment Card Industry Data Security Standard — compliance framework for organizations handling credit card data
Networking
OSI Model7-layer conceptual model for network communication — Physical, Data Link, Network, Transport, Session, Presentation, Application
TCP/IPTransmission Control Protocol / Internet Protocol — the fundamental communication protocols of the internet
DNSDomain Name System — translates human-readable domain names (google.com) into IP addresses (142.250.80.46)
CIDRClassless Inter-Domain Routing — notation for IP address ranges (e.g., 10.0.0.0/16 = 65,536 addresses)
NATNetwork Address Translation — maps private IPs to public IPs, allowing internal servers to access the internet
Reverse ProxyServer that sits in front of backend servers, forwarding client requests — used for load balancing, SSL, and caching (Nginx)
SSL/TLSSecure Sockets Layer / Transport Layer Security — encrypts data in transit between client and server (HTTPS)
SubnetA segmented portion of a larger network — used in VPCs to isolate public-facing and private resources
Version Control & Collaboration
GitDistributed version control system — tracks changes to files, enabling collaboration and history management
BranchIndependent line of development in Git — allows working on features without affecting the main codebase
Pull Request (PR)Request to merge changes from one branch into another — includes code review, discussion, and approval
Merge vs RebaseMerge creates a commit joining two branches; Rebase rewrites history to place commits on top of another branch
GitFlowBranching strategy with dedicated branches for features, releases, and hotfixes — structured but heavyweight
Trunk-BasedBranching strategy where all developers commit to a single main branch with short-lived feature branches
Formats, Protocols & General
YAMLHuman-readable data format used for configuration files — Kubernetes, Docker Compose, Ansible, and CI/CD all use YAML
JSONJavaScript Object Notation — lightweight data format for APIs and configuration (parsed with jq on command line)
APIApplication Programming Interface — defined way for software components to communicate (REST APIs use HTTP)
RESTRepresentational State Transfer — architectural style for APIs using HTTP methods (GET, POST, PUT, DELETE)
SSHSecure Shell — encrypted protocol for remote server access and command execution (port 22)
HTTP/HTTPSHypertext Transfer Protocol (Secure) — the protocol browsers use to communicate with web servers (port 80/443)
SRESite Reliability Engineering — discipline applying software engineering to operations problems (coined by Google)
Post-MortemBlameless analysis conducted after an incident — documents what happened, why, and how to prevent recurrence
